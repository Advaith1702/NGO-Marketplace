# NoSQL Document Design — NGO Surplus & Donation Marketplace

This document provides a comprehensive breakdown of the MongoDB NoSQL document structure used in the NGO Marketplace application. It covers each collection (document), its internal structure, the classes/entities it represents, and a detailed justification for every embedding and referencing decision, including denormalisation trade-offs.

---

## Overview of Collections

The application uses 4 primary collections stored in MongoDB:

1. users: Stores all platform users (NGOs, Donors, and Admins) in a single polymorphic collection
2. listings: Stores surplus item listings created by NGOs for redistribution
3. requests: Stores item transfer requests made by one NGO for another NGO's listing
4. donations: Stores monetary donation transactions from Donors to NGOs

---

## 1. Users Collection

### Document Name
users

### Structure
{
  "_id": ObjectId,
  "role": String,              // enum: ["ngo", "donor", "admin"]
  "email": String,             // unique, lowercase, trimmed
  "passwordHash": String,      // bcrypt-hashed password
  "isVerified": Boolean,       // default: false (admin-controlled NGO verification)
  "isRestricted": Boolean,     // default: false (admin-controlled access restriction)
  "profileDetails": {          // EMBEDDED sub-document
    "name": String,            // required — display name of the user/organisation
    "description": String,     // default: "" — about the NGO or donor
    "registrationId": String,  // default: "" — official government registration ID for NGOs
    "contact": String,         // default: "" — phone number or contact info
    "category": String         // default: "" — operational category (e.g., "healthcare", "education")
  },
  "createdAt": Date,           // auto-generated by Mongoose timestamps
  "updatedAt": Date            // auto-generated by Mongoose timestamps
}

### Indexes
- email: unique index (enforced by unique: true in schema)
- profileDetails.name: ascending index (optimises NGO search by name)
- profileDetails.category: ascending index (optimises filtering by operational category)

### Classes / Entities Represented
This single collection serves as a polymorphic document representing three distinct user classes:
- NGO: (role: "ngo") Non-Governmental Organisations that can create surplus listings, receive donation money, make/receive item requests, and have their own public profile page. Requires admin verification.
- Donor: (role: "donor") Individual or organisational donors who can browse the marketplace, view NGO profiles, and make monetary donations to NGOs.
- Admin: (role: "admin") Platform administrators who can verify/restrict NGOs, view platform-wide analytics, and manage all users.

### Justification for Embedding / Referencing

* profileDetails — EMBEDDED (Sub-document)
Decision: Embed profile details directly inside the User document rather than creating a separate profiles collection.
Justification:
1. One-to-one relationship: Every user has exactly one profile. There is no scenario where a profile exists independently of a user or where multiple users share a single profile. Embedding is the canonical MongoDB pattern for 1:1 relationships.
2. Read locality: Profile data (name, description, category, contact) is almost always read together with the user's core data (role, email, verification status). Embedding eliminates the need for a $lookup or secondary query on every user fetch, reducing latency.
3. Atomic updates: Updates to profile information (e.g., an NGO updating their description) are guaranteed atomic within a single document write. A separate collection would require multi-document transactions for consistency.
4. Document size: Profile details are small and bounded (a few short strings). There is no risk of exceeding MongoDB's 16 MB document size limit.
5. Query efficiency: Indexes on profileDetails.name and profileDetails.category enable fast compound queries across both core user data and profile data without requiring joins.

* Polymorphic Design — Single Collection for Multiple Roles
Decision: Store NGOs, Donors, and Admins in the same users collection, differentiated by the role field, rather than creating separate ngos, donors, and admins collections.
Justification:
1. Shared authentication: All user types share the same authentication flow (email + password), verification methods, and JWT token generation. A single collection avoids duplicating auth logic across multiple collections.
2. Simplified referencing: Other collections (Listings, Requests, Donations) reference users via ObjectId. A single collection means a single ref: 'User' is sufficient, rather than needing conditional references to different collections.
3. Admin management: The admin dashboard queries all NGOs with { role: 'ngo' } and all donors with { role: 'donor' }. A single indexed role field makes these queries efficient without cross-collection lookups.
4. Sparse fields: Fields like isVerified and registrationId are only meaningful for NGOs, but they are lightweight (Boolean/String) and default to safe values. The storage overhead of these sparse fields on Donor documents is negligible compared to the architectural complexity of separate collections.

### Denormalisation
No denormalisation applied on this collection. The User document is the single source of truth for user identity and profile data. Other collections reference it via ObjectId and use Mongoose .populate() to resolve names at query time. This avoids stale data issues (e.g., if an NGO changes its name, all references automatically resolve to the updated name).

## 2. Listings Collection

### Document Name
listings

### Structure
{
  "_id": ObjectId,
  "ownerNgoId": ObjectId,      // REFERENCE → users._id (the NGO that created this listing)
  "title": String,             // required, trimmed — name of the surplus item
  "description": String,       // required — detailed description of the item
  "quantity": Number,          // default: 1 — number of units available
  "category": String,          // enum: ["food", "clothing", "medical", "education", "electronics", "furniture", "other"], default: "other"
  "urgency": String,           // enum: ["low", "medium", "high"], default: "medium"
  "status": String,            // enum: ["available", "pending", "claimed"], default: "available"
  "createdAt": Date,           // auto-generated by Mongoose timestamps
  "updatedAt": Date            // auto-generated by Mongoose timestamps
}

### Classes / Entities Represented
Listing: A single surplus item or resource that an NGO has available for redistribution. Listings flow through a lifecycle: available → pending (when a request is being reviewed) → claimed (when transfer is finalised).

### Justification for Embedding / Referencing

* ownerNgoId — REFERENCE (ObjectId → User)
Decision: Store only the ObjectId of the owning NGO, not the NGO's name or profile data.
Justification:
1. One-to-many relationship: A single NGO can own many listings (potentially hundreds). Embedding the entire NGO document inside each listing would create massive data duplication. Referencing is the standard MongoDB pattern for 1:many relationships where the "many" side is queried independently.
2. Independent access patterns: Listings are queried independently of their owners — the marketplace page fetches all available listings with filters (category, urgency), not through the NGO document. This makes a separate collection with references more efficient than embedding listings inside User documents.
3. Unbounded growth: The number of listings per NGO is unbounded and can grow indefinitely. Embedding them in the User document would risk hitting the 16 MB document size limit and would degrade write performance as the embedded array grows.
4. Data consistency: Since the NGO's name and profile can change (e.g., updating their description), referencing avoids stale data. The application uses .populate('ownerNgoId') to resolve the current NGO name at read time.

Why NOT embed listings inside the User document?
- An NGO may have hundreds of listings over time. Embedding would create an ever-growing array inside the User document.
- Marketplace search queries (filter by category, sort by date, paginate) would be extremely inefficient against a nested array compared to a top-level collection with proper indexes.
- Listing updates (status changes, quantity updates) would require complex array manipulation ($set with positional operators) instead of simple document-level updates.

### Denormalisation
No denormalisation applied. The listing document does not store the NGO's name or any profile data. Instead, the application uses Mongoose .populate('ownerNgoId', 'profileDetails.name') to resolve the owner's display name at query time.
Trade-off considered: Denormalising the NGO name into each listing (e.g., adding an ownerName field) would eliminate the populate/lookup cost on marketplace queries. However, this was rejected because:
- NGOs can update their names, and propagating name changes across potentially hundreds of listing documents would require a bulk update operation, introducing consistency risks.
- The populate cost is acceptable for the current scale (hundreds to low thousands of listings).
- Marketplace queries already index on category and status, so the additional populate for name resolution is a marginal cost.

## 3. Requests Collection

### Document Name
requests

### Structure
{
  "_id": ObjectId,
  "listingId": ObjectId,         // REFERENCE → listings._id (the listing being requested)
  "requestingNgoId": ObjectId,   // REFERENCE → users._id (the NGO making the request)
  "message": String,             // default: "" — optional message explaining the request
  "status": String,              // enum: ["pending", "accepted", "rejected"], default: "pending"
  "createdAt": Date,             // auto-generated by Mongoose timestamps
  "updatedAt": Date              // auto-generated by Mongoose timestamps
}

### Classes / Entities Represented
Request: Represents an item transfer request from one NGO (the requester) to another NGO (the listing owner). Acts as a junction/bridge entity in the many-to-many relationship between NGOs and Listings. Follows a workflow: pending → accepted or rejected.

### Justification for Embedding / Referencing

* listingId — REFERENCE (ObjectId → Listing)
Decision: Reference the listing by ObjectId rather than embedding listing data.
Justification:
1. Many-to-many relationship: A single listing can receive multiple requests from different NGOs, and a single NGO can request multiple different listings. This many-to-many relationship is best modelled as a separate junction collection with references to both sides.
2. Independent lifecycle: Requests have their own status workflow (pending → accepted/rejected) that is independent of the listing's lifecycle. A separate document allows clean status management without entangling it with the listing's data.
3. Query flexibility: The owning NGO needs to query "all requests for my listings" while the requesting NGO needs to query "all requests I have made." A separate collection with indexed references supports both access patterns efficiently.

* requestingNgoId — REFERENCE (ObjectId → User)
Decision: Reference the requesting NGO by ObjectId.
Justification:
1. Same rationale as ownerNgoId in Listings: The requesting NGO's profile data may change, and referencing avoids stale data.
2. Query pattern: The dashboard uses .populate('requestingNgoId', 'profileDetails.name') to display the requester's name, ensuring always-current data.

Why NOT embed requests inside the Listing document?
- A listing could receive many requests (unbounded). Embedding would grow the Listing document unpredictably.
- Request status updates would require array positional updates ($elemMatch + $set) instead of simple document updates.
- The requesting NGO's dashboard needs to query their outgoing requests across all listings — this is extremely inefficient with an embedded array (requires scanning all Listing documents).

### Denormalisation
No denormalisation applied. Neither the listing title nor the NGO name is stored in the Request document. Both are resolved at read time using .populate().
Trade-off: This means displaying a request on the dashboard requires populating two references (listing + requesting NGO). At the current scale, this is acceptable. If the platform scaled to millions of requests, denormalising listingTitle and requestingNgoName into the Request document would improve read performance at the cost of requiring update propagation when source data changes.

## 4. Donations Collection

### Document Name
donations

### Structure
{
  "_id": ObjectId,
  "donorId": ObjectId,       // REFERENCE → users._id (the donor who made the contribution)
  "ngoId": ObjectId,         // REFERENCE → users._id (the NGO receiving the donation)
  "amount": Number,          // required, min: 1 — monetary value of the donation
  "message": String,         // default: "" — optional message from the donor
  "status": String,          // enum: ["completed", "failed"], default: "completed"
  "createdAt": Date,         // auto-generated by Mongoose timestamps
  "updatedAt": Date          // auto-generated by Mongoose timestamps
}

### Classes / Entities Represented
Donation: Represents a single monetary donation transaction from a Donor to an NGO. Each donation is an immutable financial record once created. Used for the donations ledger, analytics dashboards, and donation trend charts.

### Justification for Embedding / Referencing

* donorId — REFERENCE (ObjectId → User)
Decision: Reference the donor by ObjectId.
Justification:
1. One-to-many relationship: A single donor can make many donations over time. Embedding donation records inside the User document would create an unbounded, ever-growing array.
2. Financial audit trail: Donations are transaction records that should exist as independent, immutable documents. They may need to be queried, aggregated, and reported on independently of the donor or NGO that created them (e.g., "total donations this month across the platform").
3. Aggregation pipelines: The analytics dashboards use MongoDB aggregation pipelines ($group, $sum, $match) on the donations collection to calculate donation trends, top NGOs by donation amount, and monthly totals. These aggregations run most efficiently on a flat, top-level collection rather than on nested arrays within User documents.

* ngoId — REFERENCE (ObjectId → User)
Decision: Reference the receiving NGO by ObjectId.
Justification:
1. Same rationale as donorId: The many-to-many relationship between donors and NGOs (a donor can give to many NGOs; an NGO can receive from many donors) is best modelled through a separate junction collection.
2. Analytics access pattern: The admin dashboard's "NGO with most donations" metric aggregates by ngoId using $group. A referenced ObjectId in a flat collection is the optimal structure for this aggregation.
3. NGO dashboard: Each NGO's dashboard uses { ngoId: req.user._id } to fetch their incoming donations. A top-level collection with an indexed ngoId field supports this query efficiently.

Why NOT embed donations inside the User document?
- Unbounded growth: A popular NGO could receive thousands of donations. Embedding would bloat the User document toward the 16 MB limit.
- Dual ownership: Each donation belongs to both a Donor and an NGO. Embedding in the Donor document would make it invisible to the NGO, and vice versa. Duplication would create consistency nightmares.
- Aggregation inefficiency: Platform-wide analytics (monthly trends, total amounts) would require $unwind operations on arrays across all User documents, which is orders of magnitude slower than aggregating a flat collection.

### Denormalisation
No denormalisation applied. The donation document does not store the donor's name or the NGO's name. Both are resolved via .populate() at read time.
Trade-off justification: Donations are primarily displayed in two contexts:
1. Donations ledger (Donor/NGO dashboard) — small result sets per user, populate cost is minimal.
2. Analytics aggregation — these pipelines group by ObjectId and sum amounts; they do not need display names. The top NGO's name is resolved with a single findById() after aggregation.
If the platform scaled significantly, denormalising donorName and ngoName would speed up ledger rendering but would require propagation logic when users update their profiles.

│       USERS        │

│                    │
│ profileDetails     │
│   [EMBEDDED]       │
└──────┬───┬───┬─────┘
       │   │   │
       │   │   └──────── donorId (ref) ────┐
       │   │                               │
       │   └──── ownerNgoId (ref) ──┐      │
       │                            │      │
       │   requestingNgoId (ref)    │      │
       │         │                  │      │
       ▼         ▼                  ▼      ▼
┌──────────┐  ┌──────────┐  ┌──────────────────┐
│ REQUESTS │  │ LISTINGS │  │    DONATIONS     │
│          │──│          │  │                  │
│listingId │  │          │  │ ngoId (ref→User) │
│  (ref)   │  │          │  │donorId(ref→User) │
└──────────┘  └──────────┘  └──────────────────┘

All inter-document relationships use REFERENCING (ObjectId).
The only EMBEDDING is profileDetails within the User document.
